 fork MAA_MHXY_MG, 然后做迁移 MYscript 的特性 MAA_MHXY_MG 做时空版的适配呢？时空版也是手游，区别是 MAA_MHXY_MG 操作的是安卓模拟器里的手游，时空版是官方出的客户端
时空版 UI 和手游版是一致的
## Fork MAA_MHXY_MG + 时空版适配方案分析

这个思路**非常务实**，比"两个项目全量合并"靠谱得多——有明确的主干项目、明确的目标游戏、特性是增量叠加，不是推倒重来的架构重构。

---

### 一、先确认核心前提

#### 时空版和手游版的关系
- **游戏内容高度一致**：梦幻西游：时空本质上是官方 PC 端，玩法、任务流程、UI 布局和手游版基本相同
- **渲染方式不同**：手游版是安卓模拟器（ADB 控制），时空版是原生 Windows 客户端（Win32 窗口）
- **分辨率/UI 缩放可能有差异**：时空版有 PC 端的窗口缩放逻辑，模板图不能直接复用

#### MaaFramework 的 Win32 支持
MAA 框架本身就支持 **Win32 Controller**，不是只能用 ADB。截图、点击、输入这些底层能力对 Windows 窗口原生支持——这是整个方案成立的技术基础。

---

### 二、方案整体架构

```
Fork 自 gitlihang/Maa_MHXY_MG
        │
        ├── 第一层：Controller 适配（ADB → Win32）
        │
        ├── 第二层：资源替换（手游模板 → 时空版模板）
        │
        └── 第三层：特性增强（从 MYscript 迁移优秀设计）
                ├── 动态标定系统
                ├── 多开轮转引擎
                ├── 三重急停 + 演练模式
                ├── 组队握手机制
                └── GUI 优化
```

### 三、分阶段工作量拆解

#### 第一阶段：Win32 适配 + 基础运行验证（1-2 周）

| 工作项 | 难度 | 说明 |
|--------|------|------|
| Controller 切换 | ★☆☆ | MAA 原生支持 Win32，改配置即可，核心代码不用动 |
| 窗口连接方式 | ★★☆ | 从 ADB 连接改为窗口句柄/进程名匹配，参考 MYscript 的进程过滤逻辑 |
| 分辨率适配 | ★★☆ | 时空版默认分辨率和模拟器不同，需要调整 ROI 基准 |
| 单任务 PoC | ★★☆ | 选最简单的"签到"任务跑通，验证整条链路 |

**关键风险**：几乎没有。MAA 的 Win32 Controller 是成熟功能，这一步主要是配置调整。

#### 第二阶段：核心任务资源迁移（2-4 周）

| 工作项 | 难度 | 说明 |
|--------|------|------|
| 模板图重新截取 | ★★☆ | 每个任务的识别模板都要在时空版客户端重新截一遍 |
| ROI 坐标调整 | ★★☆ | UI 布局微调导致的坐标偏移，逐任务校准 |
| 师门任务适配 | ★★★ | 核心日常，步骤多，需要逐步骤验证 |
| 秘境/宝图/运镖适配 | ★★★ | 每个任务 1-2 天标定+测试 |
| 副本任务适配 | ★★★☆ | 320/520 副本流程长，状态多 |

**关键因素**：**时空版和手游版的 UI 相似度决定了这一步的工作量**。
- 如果 UI 90% 一致，只是分辨率缩放 → 2 周搞定
- 如果 PC 版有专门的 UI 重排 → 4 周甚至更久

#### 第三阶段：MYscript 特性迁移（3-5 周）

按优先级排序，**不是所有特性都值得迁**：

| 特性 | 优先级 | 迁移难度 | 实现方式 |
|------|--------|---------|---------|
| **多开轮转引擎** | ⭐⭐⭐⭐⭐ 最高 | ★★★ 中 | 在 MAA 外层用 Python 管理多个 MaaTasker 实例，复用 MYscript 的轮转调度逻辑 |
| **动态标定工具** | ⭐⭐⭐⭐ 高 | ★★★☆ 中高 | 独立开发标定 GUI，输出 MAA 格式的资源文件，覆盖官方默认资源 |
| **三重急停机制** | ⭐⭐⭐⭐ 高 | ★★☆ 中低 | 全局热键 + 鼠标甩屏角检测，调用 MaaTasker.stop() |
| **演练模式** | ⭐⭐⭐ 中 | ★★★ 中 | 封装一层 action 拦截器，dry_run 模式下跳过点击只打日志 |
| **组队握手** | ⭐⭐⭐ 中 | ★★★☆ 中高 | 多实例间通过文件/管道通信，队长发信号队员响应 |
| **拟人化输入** | ⭐⭐ 低 | ★☆☆ 低 | MAA click 本身就有随机偏移参数，配置一下就行 |
| **放大镜标定 ROI** | ⭐⭐ 低 | ★★☆ 中低 | 标定工具的细节优化，非核心 |

#### 第四阶段：GUI 整合 + 打包发布（2-3 周）

| 方案 | 工作量 | 说明 |
|------|--------|------|
| **用 MFAAvalonia + 插件** | 2 周 | 基于 MAA 官方 UI，把多开、标定等做成插件 |
| **移植 MYscript 的 customtkinter GUI** | 3 周 | 体验更统一，但对接 MAA API 工作量大 |

#### 总计：8-14 周（2-3.5 个月）单人全职

### 四、这个方案的核心优势

#### 1. 风险可控，可分阶段交付
- 第 2 周就能跑通签到 → 有基础可用版本
- 第 4 周核心日常齐全 → 能满足大部分需求
- 第 8 周转完 MYscript 核心特性 → 体验超越两个原始项目
- 随时可以停，每一步都有产出

#### 2. 享受 MAA 生态的持续红利
- MaaFramework 本身在持续更新，识别算法、性能优化直接受益
- 未来 MAA 出了新功能（比如更好的 OCR、神经网络识别），自动升级
- 社区资源、文档、问题排查方案都能复用

#### 3. 任务逻辑复用率高
- 只要时空版和手游版任务流程一致，Pipeline JSON 的**逻辑结构几乎不用改**
- 主要工作量在**重新截图和调坐标**，不是重写逻辑
- 30+ 个任务的状态机设计直接拿来用，省了最费脑子的部分

### 五、潜在坑点

#### 1. UI 差异可能比想象中大
PC 客户端经常会有专门的布局调整：
- 按钮大小、间距变化
- 某些功能入口位置不同
- 弹窗样式不一致

**应对**：先拿 2-3 个核心任务做 PoC，评估差异程度，再决定整体投入。

#### 2. Win32 输入权限问题
游戏客户端如果以管理员权限运行，MAA 也需要管理员权限才能注入输入——这一点 MYscript 已经踩过坑，直接复用它的提权逻辑就行。

#### 3. 动态标定和 MAA 资源体系的摩擦
MAA 的资源是按目录结构组织的，动态标定生成的模板要能正确加载进去。需要设计好**官方资源 + 用户标定资源**的优先级和覆盖规则。

#### 4. 多开的性能开销
每个号一个 MaaTasker 实例，内存占用会比 MYscript 的单进程多线程高。但对现代电脑来说，5 开以内问题不大。

### 七、落地建议

**如果决定做，按这个顺序最稳妥：**

1. **先做 1 周技术验证**：搭环境、切 Win32 Controller、拿签到任务跑通，确认时空版 UI 差异在可接受范围内
2. **再迁核心日常**：师门、秘境、宝图、运镖这四个，做完就有实用价值
3. **然后加多开轮转**：这是 MYscript 最有价值的特性，也是时空版五开玩家的刚需
4. **最后补标定工具和 GUI 优化**：这些是体验提升，不影响核心功能

**最关键的判断点**：第一周验证完，如果时空版 UI 和手游版差异很大（比如 PC 版重做了界面），那任务适配的工作量会翻倍，需要重新评估投入产出比。如果 UI 基本一致只是缩放，那这个方案性价比极高。


## UI 一致的情况下，方案可行性大幅提升

既然时空版和手游版 UI 完全一致，**适配工作量直接砍掉 60% 以上**——最大的不确定性（UI 差异）消除了，剩下的都是确定性工作。

### 三、推荐的技术路线细节

#### 项目结构设计

```
Maa_MHXY_SK（时空版 fork 项目）
├── pipeline/              # 直接复用 MAA_MHXY_MG 的任务流水线
├── resource/
│   ├── base/             # 官方模板图（从手游版缩放适配）
│   └── user/             # 用户标定覆盖资源（MYscript 特性）
├── controller/
│   └── win32_adapter.py  # Win32 Controller 封装 + 进程过滤
├── multi_instance/       # 多开轮转引擎（从 MYscript 迁移）
│   ├── rotation.py
│   └── teaming.py
├── calibrator/           # 动态标定工具（从 MYscript 迁移）
├── safety/               # 急停 + 演练模式（从 MYscript 迁移）
└── gui/                  # GUI 整合
```

#### 核心设计原则

1. **Pipeline 不动**：MAA_MHXY_MG 的任务 JSON 原封不动用，保证稳定性
2. **资源双层结构**：官方 base + 用户标定 user，用户层优先覆盖
3. **增强全部外置**：多开、标定、急停都做成外层封装，不侵入 MAA 核心
4. **渐进式迁移**：先能用 → 再加多开 → 再加标定 → 最后优化 GUI

### 四、最终结论

**UI 一致的前提下，这个方案可行性非常高，性价比拉满。**

- **1 周**：就能有一个"能跑日常"的基础版本
- **1 个月**：核心功能齐全 + 多开轮转，超越两个原始项目
- **2 个月**：特性全部迁移完，体验完整

本质上这不是"重新开发"，而是**换个 Controller + 做一层特性封装**——核心的任务逻辑和识别能力都是 MAA_MHXY_MG 已经验证过的，风险极低。

如果要动手，建议第一周先把 Win32 适配 + 师门任务跑通，验证完这条路没问题再全力投入。
